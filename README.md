# From Big Prompts to Reliable React Agents

Five sibling apps take the **same** product feature —
a support "Triage Copilot" — from a hand-rolled loop to a reliable, evaluated agent.

> The thesis: **a coding agent can write the code; the engineer designs the agent's
> _operating system_ — its instructions, tools, contracts, procedures, evals, and
> safe UI states.** Each app isolates one of those pieces.

- ✅ **`master`** (this branch) — the **finished reference**: every app complete and
  runnable. Read it, run it.
- 🌱 **`todo`** — the **starter**: the same apps with the engineer-owned pieces carved
  out into guided `WORKSHOP TODO` stubs, so you build them yourself (see `WORKSHOP.md`
  on that branch). `git checkout todo` to do the workshop.

---

## The arc — one app per beat

| ⏱️      | App                                                        | The beat                                  | What it demonstrates                                                                           |
| ------- | ---------------------------------------------------------- | ----------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 30–50   | [`tiny-agent`](./tiny-agent)                               | The smallest agent, **by hand**           | an agent is just a loop around an LLM call + your code executing tools                         |
| 50–90   | [`issue-triage`](./issue-triage)                           | First real Next.js feature + contract     | the AI SDK deletes the plumbing; a Zod **contract** + validation make output **safe UI state** |
| 100–120 | [`issue-triage-2x`](./issue-triage-2x)                     | Scale it the naive way → **drift**        | one giant system prompt buries the rules; the model drops them under load                      |
| 120–150 | [`issue-triage-skills`](./issue-triage-skills)             | **Agent Skills** / progressive disclosure | a thin prompt + load only the runbook you need → small context, rules hold                     |
| 150–170 | [`issue-triage-skills-evals`](./issue-triage-skills-evals) | **Evals**                                 | grade the agent like a product: deterministic checks + an LLM-as-judge, run repeatedly         |

The running example carries five planted "canary" rules **R1–R5** (a `Refs:` footer,
plain prose, an SLA-in-minutes, a refund-over-$50 guard, a sign-off). Watching whether
the agent keeps or drops them is how each beat makes reliability concrete.

## Setup

Requires Node 20+ and an OpenAI API key. Each app is self-contained — install and run
it from its own directory:

```bash
cd <app>
npm install
cp .env.example .env.local     # then add your OPENAI_API_KEY
```

`OPENAI_MODEL` is overridable (defaults to `gpt-5-mini` in the Next apps); the
`tiny-agent` reads `OPENAI_API_KEY` straight from your shell env.

## The apps

### 🧩 `tiny-agent` — an agent is a loop

A ~380-line TypeScript CLI hitting the raw OpenAI Responses API via `fetch`: one tool,
a hand-written model → tool → result loop with a hard step cap. No framework — so you
see exactly what every later demo's SDK is doing for you.

```bash
cd tiny-agent && npm install
npm run agent                          # built-in sample question
npm run agent -- "What happens at minute 75?"
```

### 🎛️📐🚦 `issue-triage` — first real feature, with a contract

Next.js (App Router) + React 19 + **Vercel AI SDK v6**. The same loop as `tiny-agent`,
now a `ToolLoopAgent` with three tools, streamed to a `useChat` UI — the SDK owns the
loop, the wire format, and the tool plumbing. Reliability moves to where the model's
output becomes UI state: a **Zod contract** (`lib/triage/schema.ts`) validated at the
boundary (`lib/triage/parse.ts`) so the UI only ever renders a known-good shape, with an
honest error state otherwise.

```bash
cd issue-triage && npm install && npm run dev      # http://localhost:3000
```

### 🧭 `issue-triage-2x` — the cautionary "before"

A multi-turn chat that crams the **entire** knowledge base — 18 runbooks + 6 policies +
a 180-entry incident archive (~14k tokens) — into one system prompt, re-sent every turn.
A `<ContextMeter>` shows the bloat. Push it (a big queue, low reasoning effort) and watch
it **drift**: dropped rules, the wrong runbook, contradictions. Set
`OPENAI_REASONING_EFFORT=minimal` in `.env.local` to force it.

```bash
cd issue-triage-2x && npm install && npm run dev
```

### 🗂️ `issue-triage-skills` — progressive disclosure

The fix. Same knowledge, same rules — but the prompt stays **thin** and the agent loads
only the 1–2 runbooks it needs on demand, via **Agent Skills** (Vercel's `bash-tool`,
local `just-bash` backend — no account). Each runbook is its own `skills/<id>/SKILL.md`
(generated by `npm run build:skills`); the always-on index is just one line per skill.
The incident archive sits on disk for the `bash` tool to grep. The activity log shows
_which_ runbook it loaded; the context meter drops from ~14k to ~3–4k and the rules hold.

📝 NB: In reality, you could equally reach out for retrieval tools here. This workshop focuses on "navigation" as retrieval - no vector DBs, lower cost, the agent simply navigates the file system to search and retrieve relevant information

```bash
cd issue-triage-skills && npm install
npm run build:skills      # generate skills/ from the runbook corpus
npm run dev               # http://localhost:3000
```

### 🧪 `issue-triage-skills-evals` — test the agent like a product

The skills agent plus an **[evalite](https://evalite.dev)** eval set grading it against
the R1–R5 contract. `evals/scorers.ts` mixes cheap **deterministic** checks (format rules
are just code) with one **LLM-as-judge** (is the SLA the _right number_?). `evals/triage.eval.ts`
is the dataset of `{ input, expected }` cases. The finale is an A/B: flip
`OPENAI_REASONING_EFFORT=minimal` and watch the judge catch the drift while the
deterministic checks stay green.

```bash
cd issue-triage-skills-evals && npm install
npm run eval        # evalite watch + dashboard → http://localhost:3006
npm run eval:run    # run once, print the table, exit (CI)
```

> Evals make **real OpenAI calls** (the agent runs, and the judge grades).
